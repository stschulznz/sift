### 12.3 Parallelisation opportunities

The chain has three independent prefixes that can run concurrently:

- **Config / state / audit / cost** (steps 2-5): no cross-dependencies, can be
  picked up by four agents in parallel after step 1 lands.
- **LLM stack** (steps 6-8): can run in parallel with the Graph stack
  (steps 9-11) once steps 2 and 4 are done.
- **Pipeline layers** (steps 12-15): rule layer, sender cache, classifier
  wrapper, and reference keeper are independent of each other and only
  share dependencies on earlier layers; they can be parallelised behind
  step 11.

The orchestrator (step 16) is the first hard fan-in point and must wait for
all four pipeline layers and the Graph action layer.

### 12.4 Definition of done per step

Every step in the build sequence is "done" when, in order:

1. The files listed in the table exist with the contracts from sections 4-6.
2. Unit / integration tests for that step are present and pass under
   `pytest -q`.
3. `ruff check .` and `mypy src/sift` pass with zero errors and zero
   warnings on the touched paths.
4. `python -m sift doctor` (where applicable) reports a green status for
   the new component.
5. The matching sub-issue from Section 13 is closed with a completion
   comment matching the repo's Issue Completion Protocol.

## 13. Implementation Sub-Issues

Each sub-issue maps to one row in the build sequence (12.2). Issues are
intentionally small enough to fit in a single agent session against the

### Issue #1 -- Project scaffold and CI baseline

**Goal:** A `pip install -e .` installable empty package with passing CI.

**Scope:**

  `ruff check .`, `mypy src/sift`, `pytest -q` on push and PR against

**Out of scope:** any business logic, any CLI commands.

**Acceptance:**

- `pip install -e .[dev]` succeeds on a clean venv.
- `python -c "import sift; print(sift.__version__)"` prints a non-empty

**Acceptance:**

## 14. Anti-Patterns

The patterns below are explicit non-goals. Every one of them was
considered during spec authoring and rejected for a concrete reason.
They are listed here so future contributors (human and agent) can
recognise them and not re-introduce them.

### Anti-pattern 1: LLM-driven move execution

**The temptation:** Hand the LLM the Graph token and tool schemas and
let it call `move_message` directly.

**Why it is wrong:** A single bad classification or prompt-injected
message can move thousands of items irrecoverably. The LLM's job is
classification only; the orchestrator owns dispatch. This is the
same trust-boundary lesson Solace's SR-03 enforces.

**The correct shape:** Classifier returns a typed
`ClassificationResult`. Orchestrator decides what to do with it.
Apply layer translates decision into Graph calls.

### Anti-pattern 2: Hard-deleting in v0.1.0

**The temptation:** "ToBeDeleted" obviously means "delete it."

**Why it is wrong:** A regression in classification on a 1M-message
inbox is a recoverable disaster only if everything is recoverable.
Soft-delete to a `Sift/Quarantine` folder leaves recovery one
`sift undo` away. v1.0.0 may add a gated hard-delete after a
quarantine retention period; v0.1.0 must not.

**The correct shape:** Soft-delete to quarantine. Period.
`HardDeleteForbiddenError` is a real exception in `util/errors.py`.

### Anti-pattern 3: Stateless re-classification on every run

**The temptation:** Just classify everything every time. Stateless
is simple.

**Why it is wrong:** Repeated LLM cost on the same messages, no
auditability of when a decision was made or by which model, and no
way to honour the sender-cache short-circuit.

**The correct shape:** `messages` table records every classification
decision keyed by Graph message ID. The orchestrator skips messages
already classified at the same or higher confidence with the same
ruleset hash.

### Anti-pattern 4: Logging message bodies for debugging

**The temptation:** "Just include the body in the log when
classification confidence is low, so I can debug."

**Why it is wrong:** SR-05 is unconditional. Personal mail, financial
records, medical correspondence, and authentication codes all flow
through the inbox. The audit log lives on disk, often replicated to
journald, often shipped to log aggregators. Bodies in audit are a
breach waiting to happen.

**The correct shape:** Log subject, sender, message ID, decision,
classifier model, confidence, reasoning summary (max 200 chars,
redacted). Never the body.

### Anti-pattern 5: One global LLM client shared by every provider

**The temptation:** Build one `httpx.AsyncClient` and pass it
everywhere.

**Why it is wrong:** Provider-specific defaults (timeouts, retry
budgets, headers, base URLs) cannot be expressed cleanly. Failures
on one provider leak connection state to another.

**The correct shape:** Each provider adapter owns its own
`httpx.AsyncClient` with provider-specific defaults, instantiated
once per process and reused. The router holds the registry.

### Anti-pattern 6: Synchronous "try every provider until one
works" inside a request

**The temptation:** Naive failover loop with no scoring.

**Why it is wrong:** A flapping provider gets retried on every
request, burning latency and budget before the loop falls through.

**The correct shape:** EMA-based health scoring per provider
(Section 6 smart-router). A provider that just failed sees its
score drop and is deprioritised on the next request. Recovery is
gradual.

### Anti-pattern 7: Putting credentials in the YAML config

**The temptation:** "Just put the API key right in the config file
so I do not have to set environment variables."

**Why it is wrong:** Configs get committed to git, screenshared,
emailed for support, and copied into Discord. Credentials get
revoked when this happens.

**The correct shape:** Config file references env vars or keyring
entries by name (Section 5.2 `${SIFT_*}` interpolation). Live
secrets live in env vars, OS keyring, or a secrets manager only.

### Anti-pattern 8: Using the rules layer as the primary
classification mechanism

**The temptation:** Write a rule for every sender. The LLM is
expensive.

**Why it is wrong:** Rules ossify. They drift from intent. They miss
the long tail. The whole point of the LLM layer is to handle the
unbounded shapes that rules cannot.

**The correct shape:** Rules are for the sharp-edged minority:
"never delete from $employer.com," "always Action for $manager,"
"skip pipeline for $list-id." The LLM handles everything else,
backed by the sender cache for repeat-sender amortisation.

### Anti-pattern 9: Re-classifying when the model changes silently

**The temptation:** Upgrade the configured model, ship it, hope for
the best.

**Why it is wrong:** A model change can shift the entire decision
distribution. Yesterday's `Action` might become tomorrow's
`ReadLater`. Users who have built up trust in the system see
silent regressions.

**The correct shape:** The `messages` table records the model that
made each decision. A model change shows in the next `sift doctor`
run as a "model drift detected" advisory, not a silent re-tag.
Re-classification is opt-in via `sift backfill --reclassify`.

### Anti-pattern 10: Building a web UI in v0.1.0

**The temptation:** "It would be so much nicer with a dashboard."

**Why it is wrong:** A web UI is a separate project: auth, hosting,
CSRF, sessions, deployment. v0.1.0 is a CLI-first tool that does
its job from cron / systemd / a scheduler. A dashboard is a
post-1.0 addition (likely a separate `sift-ui` repo, with `sift`
remaining the headless engine).

**The correct shape:** CLI + JSONL audit + structured logs. Anyone
who wants a dashboard can point Grafana / Loki at the audit log.

## 15. Decision Log

| Decision | Alternatives considered | Rationale |
|----------|-------------------------|-----------|
| **CLI-first, no web UI in v0.1.0** | Web UI, TUI, both | Solo maintainer, zero hosting story, fastest path to value. A UI is a separate project. |
| **Microsoft Graph only in v0.1.0** | IMAP, JMAP, multi-protocol from day one | The triggering use case is a 1M-message M365 inbox. IMAP + JMAP are tier-2 work for v0.2 once the Graph path is solid. |
| **SQLite for state** | Postgres, JSON files, no persistent state | One-file deploy, zero ops, durable. Postgres is overkill for a per-user CLI. JSON files lose ACID on crash. |
| **Vendored llm-router fork, not git submodule or runtime dep** | Submodule, PyPI dep on Solace router, fresh router | Solace is private and tightly coupled to its own state model. Vendoring a slimmed-down copy gives Sift independent release cadence and an OSS-clean dep tree. |
| **Pydantic v2 strict mode everywhere** | dataclasses, attrs, runtime-typed dicts | Strict catches contract drift between the YAML config, the Graph wire format, and the LLM JSON envelopes. The cost is a few `model_config = ConfigDict(strict=True)` lines. |
| **httpx with explicit per-adapter clients** | requests, aiohttp, one shared client | httpx is async-native, supports HTTP/2, and matches the Solace router's choice. Per-adapter clients keep failures isolated (anti-pattern 5). |
| **structlog for logging** | stdlib logging, loguru | structlog renders the same JSON shape that the audit log expects, and the redaction middleware applies uniformly to console + audit + journald. |
| **click for CLI** | argparse, typer, fire | click is mature, has rich subcommand support, and produces helpful auto-generated `--help` output. Typer is built on click; using click directly removes a layer. |
| **Provider priority: Copilot, Claude, Azure OpenAI, MS Foundry first** | OpenAI direct, OpenRouter, Ollama in v0.1.0 | The maintainer already pays for Copilot; classification is cheap there. Claude and Azure are proven on the equivalent Solace workload. Ollama and OpenRouter slip to v0.2 where local-first and multi-provider value-prop matter more. |
| **EMA-based smart routing, not simple round-robin** | Round-robin, random, sticky | A flapping provider should hurt its own score and recover gradually. Round-robin amplifies flap; sticky leaves users on a degraded provider. |
| **Soft-delete to quarantine, never hard-delete** | Hard-delete with retention period, two-mode soft + hard | Recoverability beats elegance for v0.1.0 (anti-pattern 2). Quarantine is one folder, one undo, one expiry rule. |
| **JSONL audit, not a database table** | Audit table in SQLite, structured DB, no audit | JSONL is grep-friendly, ships cleanly to log aggregators, and survives DB corruption. The cost is no SQL queries on audit; that is fine because audit is for forensics, not OLAP. |
| **No telemetry / phone-home in v0.1.0** | Anonymous opt-in, opt-out, on by default | OSS trust is built by not collecting anything. v1.0 may add an opt-in `sift telemetry enable` that ships only aggregate counts. |
| **Single-user per install** | Multi-tenant, per-user isolation in one process | Sift is a personal tool. Multi-tenant adds auth, isolation, and shared-state complexity for zero v0.1.0 value. Two users run two installs. |
| **PyPI distribution under `sift-mail`** | `sift`, `sift365`, `siftbox`, `siftmail` | `sift` on PyPI is taken; `sift-mail` is descriptive and available. The console script remains `sift`. |
| **Python 3.13+ floor** | 3.11, 3.12 | Solo maintainer; no need to support older Pythons. 3.13's improved error messages and per-interpreter GIL story are genuinely useful. |
| **MIT license** | Apache-2.0, AGPL, BSL | MIT is the lowest-friction OSS license for adoption. Apache adds an explicit patent grant that is overkill for a single-author project. AGPL / BSL would block most contributors. |

## 16. Glossary

| Term | Definition |
|------|------------|
| **Sift** | The project itself: an OSS tool that classifies a Microsoft 365 inbox into Action / Read / ToBeDeleted / WaitingFor folders using LLM-backed classification. |
| **Inbox triage** | The general problem domain: deciding, for each incoming message, what kind of attention (if any) it deserves. |
| **Pipeline** | The per-message processing chain: rules -> sender cache -> LLM classifier -> reference keeper -> apply. |
| **Apply** | The final pipeline stage that translates a classification decision into Microsoft Graph actions (move, mark-read, soft-delete). |
| **Classification** | The act of assigning one of the five canonical decisions to a message: `Action`, `ReadLater`, `ToBeDeleted`, `WaitingFor`, or `quarantine_for_review`. |
| **Decision** | A typed `ClassificationResult` produced by the classifier (or by an earlier pipeline stage that short-circuits the classifier). |
| **Quarantine** | A configured folder (default: `Sift/Quarantine`) where messages with low-confidence or malformed classifications land for human review. Also where soft-deleted messages live. |
| **Soft-delete** | Moving a message to the quarantine folder rather than invoking Microsoft Graph's hard-delete. The only "delete" verb Sift v0.1.0 supports. |
| **Hard-delete** | An irreversible removal via Microsoft Graph's `DELETE /messages/{id}`. Forbidden in v0.1.0; raises `HardDeleteForbiddenError`. |
| **Reference keeper** | The safety-net pipeline stage that overrides any `ToBeDeleted` decision when the message matches a "never delete" rule. |
| **Backfill** | A one-shot run that processes a date-bounded historical mailbox slice rather than the live unread queue. Used to clean up the existing 1M-message inbox. |
| **Sender cache** | A per-sender memo of recent classification decisions used to short-circuit the LLM classifier when a sender's recent pattern is consistent and high-confidence. |
| **Rule layer** | Deterministic, ordered rules evaluated before the LLM classifier. Used for sharp-edged cases ("never delete from employer.com"). |
| **LLM router** | The vendored, slimmed-down adapter layer that fronts the four LLM providers and handles failover, scoring, and cost tracking. |
| **Provider** | A concrete LLM backend Sift can call: `github_copilot`, `claude`, `azure_openai`, `microsoft_foundry`. |
| **Failover** | The smart-router behaviour of trying providers in scored priority order, recording health per provider, and recovering gradually after a failure. |
| **Smart routing** | EMA-based scoring of provider health used to drive failover decisions. |
| **Cost tracker** | The component that records per-LLM-call cost into the `cost_log` table and produces per-run aggregates. Returns `NULL` (not zero) when pricing for a `(provider, model)` is unknown. |
| **Audit log** | An append-only JSONL stream of every consequential action Sift takes (classification, move, undo, override). Subject only, never body content. |
| **Undo journal** | A separate append-only JSONL store recording the inverse of every Graph mutation, used by `sift undo` to reverse the most recent N actions. |
| **Doctor** | The `sift doctor` CLI command that runs readiness checks (Graph auth, providers reachable, state DB writable, config valid) and prints a status report. |
| **StateStore** | The SQLite-backed persistence layer holding `messages`, `actions`, `undo_journal`, `sender_cache`, `cost_log`, and `migrations` tables. |
| **Dry-run** | A pipeline mode (`--dry-run`) that runs every stage including classification but skips the apply layer's Graph writes. Used for safe inspection on unfamiliar mailboxes. |
| **Action / Read / ToBeDeleted / WaitingFor** | The four user-facing destination folders. The fifth decision, `quarantine_for_review`, routes to the quarantine folder. |
| **Reference-keeper rule** | A configured pattern (sender, domain, header, list-id, regex) that vetoes any `ToBeDeleted` classification on a matching message. |
| **Live suite** | Tests marked `@pytest.mark.live` that hit real Microsoft 365 and real LLM providers. Skipped by default; only run via the manual-dispatch GitHub Actions workflow against a dedicated test tenant. |
| **Vendored** | A copy of upstream code (here, the Solace LLM router) embedded in the Sift repository under `src/sift/llm/` rather than pulled at runtime via PyPI. |
| **OSS clean** | A dependency posture where every transitive dep is on PyPI with a permissive license, with no private-registry or git+https references. |
