## 7. Security Requirements

### SR-01: Graph token storage

**Threat.** Microsoft Graph delegated OAuth tokens (access and refresh) are bearer credentials that grant the holder `Mail.ReadWrite`, `MailboxSettings.Read`, `offline_access`, and `User.Read` against the user's Microsoft 365 mailbox. A token left at world-readable file permissions, embedded in a process command line, dropped into a shell history file, or written to a log allows any other local process or any operator with shell access to the host to read, move, and quarantine the user's mail. Refresh tokens are particularly sensitive because they have a long validity window (configurable by the tenant; up to 90 days by default) and can mint fresh access tokens without re-prompting the user. The threat model assumes a single-tenant host but does not assume an empty multi-user host: containers and dev hosts often have multiple humans with sudo.

**Control.** Tokens SHALL be stored in the OS keychain via `keyring` by default (macOS Keychain, Windows Credential Manager, or Secret Service / kwallet on Linux). When the OS keychain is unavailable (typical headless Linux container, no DBus session), tokens SHALL be stored at `~/.local/share/sift/.tokens` with file mode 0600 and parent directory mode 0700, both verified on every read. When neither the keychain nor a writable 0600 fallback path is available, the system SHALL fail loud at startup with exit code 3 and a diagnostic message naming the keychain backend errors and the file-system error; the system SHALL NOT silently downgrade to an insecure storage path. The fallback location is `XDG_DATA_HOME`-aware (`$XDG_DATA_HOME/sift/.tokens` when `XDG_DATA_HOME` is set).

**Verification.** A test fixture SHALL set `keyring.set_keyring(FailingKeyring())` and an unwritable target directory and SHALL assert exit code 3 and the diagnostic in stderr. A second fixture SHALL `chmod 0644` the fallback file between writes and SHALL assert that the next read fails closed. CI SHALL run `find ~/.local/share/sift -type f -perm /o=rwx,g=rwx` (or PowerShell equivalent on Windows) on the test scratch directory and SHALL assert zero matches. A static check (`ripgrep`) SHALL fail the build if any source file references a token path without invoking the permission helper `sift.security.permissions.assert_locked_down`.

### SR-02: Provider API key storage

**Threat.** Provider API keys (`ANTHROPIC_API_KEY`, `AZURE_OPENAI_API_KEY`, the Foundry client secret, the GitHub Copilot `gho_*` token covered separately under SR-03) are bearer credentials. A key committed to the repo, embedded in a YAML config that gets shared in a bug report, or echoed into a stack trace lets the holder consume the user's quota and accrue billing under the user's account. Pydantic's default `__repr__` happily prints field values; an exception that surfaces a settings model in a traceback can leak every key in a single line. Structured loggers can serialise model instances by accident inside `extra={"settings": cfg}` invocations.

**Control.** Provider API keys SHALL be sourced exclusively from environment variables, the OS keychain, or a `.env` file at file mode 0600. The configuration loader (`sift.config.Settings`, see chunk 2 component spec) SHALL declare every secret field as `pydantic.SecretStr` with `Field(repr=False)` so that neither `repr()` nor `model_dump()` (default `exclude_none=False`) leaks the value. The structured logger SHALL register a `structlog` processor `redact_secrets` that walks event-dict values, identifies `SecretStr` instances by isinstance check, and replaces them with `[REDACTED:<field_name>]`. YAML loading SHALL reject any document that contains a key matching `*_api_key`, `*_secret`, `*_token`, or `*_password` at any nesting depth, and SHALL exit with status code 1 and a remediation hint pointing to environment variables.

**Verification.** A unit test SHALL load a YAML containing `llm: {api_key: "x"}` and SHALL assert a `ConfigValidationError` with message text matching "API keys SHALL NOT be present in YAML"; the assertion SHALL match exit code 1. A second test SHALL construct a `Settings` instance with a populated `SecretStr` and SHALL assert that `repr(settings)` does not contain the secret string. A third test SHALL run a structlog event through the redactor with a `SecretStr` value in `extra` and SHALL assert the output JSON line contains `[REDACTED:provider_key]`. CI SHALL run `gitleaks` against the repo on every PR and SHALL fail the build on any positive match.

### SR-03: GitHub Copilot token storage

**Threat.** The GitHub Copilot `gho_*` token authorises Copilot model access and, depending on the user's GitHub plan, can also expose repository scope. The token has a refresh window measured in hours, not days, but a leaked active token plus an unrevoked device code lets an attacker pull fresh tokens. Sift's Copilot adapter (vendored from Project Solace) supports auto-discovery of the user's existing Copilot CLI cache so that users do not have to re-run device-code flow; that auto-discovery is read-only and does not attempt to write into another tool's cache directory. Sift's own cache is a separate, write-controlled file.

**Control.** Auto-discovery SHALL search XDG paths in this fixed order, read-only, and SHALL stop at the first hit: `$XDG_CONFIG_HOME/github-copilot/apps.json`, `~/.config/github-copilot/apps.json`, `~/Library/Application Support/github-copilot/apps.json` (macOS), and `%LOCALAPPDATA%\github-copilot\apps.json` (Windows). Sift's own cache SHALL live at `$XDG_CONFIG_HOME/sift/copilot-auth.json` (default `~/.config/sift/copilot-auth.json`) at file mode 0600 with parent directory at 0700, both verified on every read and write. The token SHALL NEVER be embedded in configuration YAML, environment defaults baked into a Docker image layer, or any process command line. Sift SHALL refuse to start when the cache file exists at a wider mode than 0600 (exit code 3, no auto-fix, the user must `chmod 0600` themselves so they observe the warning).

**Verification.** A test fixture SHALL place a fake `apps.json` at each XDG candidate path one at a time and SHALL assert the discovery helper returns the correct path and never opens the file in write mode. A second fixture SHALL `chmod 0644` `~/.config/sift/copilot-auth.json` and SHALL assert exit code 3 with the diagnostic. CI SHALL grep for any string matching `gho_[A-Za-z0-9]+` across the repo (excluding `vendor/`, the test fixture vault, and the redactor's own pattern definition) and SHALL fail on any match. A runtime assertion in the Copilot adapter SHALL panic if the token cache directory's mode is not 0700 at process start.

### SR-04: No credentials in logs

**Threat.** Even with `repr=False` on settings (SR-02), credentials can still leak into logs through three vectors: (a) HTTPX request and response logging at DEBUG level, which captures `Authorization` headers verbatim; (b) provider error responses that echo the credential back in a JSON `error.message` field; (c) human-written log statements that interpolate URLs containing query-string credentials (legacy SAS tokens, presigned URLs). A single DEBUG log enabled by an operator chasing a bug can dump every secret in the process to disk.

**Control.** A `structlog` processor `redact_credential_patterns` SHALL run as the last processor before serialisation. The processor SHALL apply, in order, the regular expressions `gho_[A-Za-z0-9_]+`, `cpt_[A-Za-z0-9_]+`, `eyJ[A-Za-z0-9_\-]{20,}\.[A-Za-z0-9_\-]{20,}\.[A-Za-z0-9_\-]+` (JWT shape), and `[A-Za-z0-9_\-]{32,}` (catch-all for opaque bearer tokens) to every string-typed value in the event dict, replacing matches with `[REDACTED]`. The same processor SHALL run inside the exception serialiser used by `structlog.processors.format_exc_info` so that traceback frames are scrubbed. HTTPX's request logger SHALL be configured at WARNING by default; users who set DEBUG SHALL receive a stderr banner at startup warning that DEBUG-level HTTP logging may capture sensitive headers and that the redactor will scrub them but operators should still avoid sharing the log file. User-visible error strings (CLI stderr, exit messages) SHALL pass through the same redactor before write.

**Verification.** A unit test SHALL log an event with an `Authorization: Bearer gho_ABC123...XYZ` header value and SHALL assert the serialised line contains `[REDACTED]` and not the original token. A second test SHALL raise an exception whose `str()` contains a 64-character hex string and SHALL assert the formatted traceback contains `[REDACTED]`. A backfill smoke test SHALL run a 1,000-message backfill against recorded fixtures, SHALL grep the resulting audit log and structured log for each of the four credential regex patterns, and SHALL assert zero matches. This satisfies success criterion 13 in chunk 1 Section 1.

### SR-05: No message bodies in audit log

**Threat.** The audit log is the system of record for what Sift moved and why, but it is also the most likely artefact to be shared (email attachment to a colleague during incident review, paste into a bug report, archival to long-term storage). A full message body in an audit log expands the blast radius of any disclosure to the entire mailbox, including legally privileged, medically sensitive, or personally identifying content the user has no contract to share. Even subject lines can carry PII (an OAuth provider's password reset email leaks the user's email and the originating service in the subject alone).

**Control.** The audit logger SHALL emit only the fields enumerated in chunk 1 FR-38: ISO-8601 timestamp, run ID, event type, message ID, sender address (clear), subject hash (SHA-256 of the subject as UTF-8 bytes, hex-encoded), classifier, confidence, source folder ID, target folder ID, provider, model, and outcome. The `Message` Pydantic model (chunk 2) SHALL declare `body`, `body_preview`, `attachments`, `recipients`, and `subject` as `Field(repr=False)` so that an accidental `logger.info("processed", message=msg)` call cannot leak them. A pre-write hook on the audit logger SHALL reject any event-dict that contains a key matching `body`, `body_preview`, `attachment`, `recipients`, or `to_recipients`, and SHALL raise a hard error at test time and a sentinel `[REJECTED]` placeholder at runtime (defence in depth; the field should never be presented to the audit logger in the first place).

**Verification.** A unit test SHALL attempt to write an audit event containing `body="hello"` and SHALL assert the helper raises `AuditPolicyError`. A second test SHALL hash a known subject string and SHALL assert the audit field matches `hashlib.sha256("...").hexdigest()`. A property-based test (`hypothesis`) SHALL generate random `Message` instances, route them through the full classification pipeline against a recorded LLM fixture, and SHALL assert that the resulting audit log lines contain none of the input bodies, none of the attachment file names, and none of the input subject strings (though the subject hash SHALL be present).

### SR-06: Least-privilege Graph scopes

**Threat.** Over-scoping the OAuth grant gives Sift permissions it does not need and asks the user to consent to capabilities the tool will never use, which is both a security failure (a Sift compromise becomes a worse compromise) and a usability failure (the consent screen becomes a wall of red flags, conditioning users to click through). `Mail.Send` would let a compromised Sift exfiltrate mail or impersonate the user; `Mail.ReadWrite.Shared` would extend reach to delegated mailboxes; any directory-level scope would expose the user's tenant directory.

**Control.** The Graph OAuth flow SHALL request exactly four scopes: `Mail.ReadWrite`, `MailboxSettings.Read`, `offline_access`, and `User.Read`. The scope list SHALL be a single named constant `sift.graph.auth.SIFT_REQUIRED_SCOPES` referenced by both the auth flow and the readiness check (chunk 2 component spec). Any change to the constant SHALL trigger a CI policy check that fails unless the change is co-located with an update to `docs/security.md` and `VENDORED.md`. The auth flow SHALL print the requested scopes to stdout at the start of `sift auth login` so the user can compare against the consent screen. The readiness check (run before the first Graph call of every Sift command) SHALL parse the cached access token's `scp` claim and SHALL refuse to start if any of the four required scopes is missing or if any unexpected scope is present (an out-of-band re-consent that elevated permissions should be visible to the operator).

**Verification.** A unit test SHALL assert the constant equals the four-element set. A second test SHALL parse a JWT containing extra scopes and SHALL assert the readiness check raises a `ScopeDriftError` and exits with code 3. A CI policy test SHALL diff the scope constant against `docs/security.md` and SHALL fail if they are out of sync. The Graph integration test SHALL record the device-code flow against a recorded fixture and SHALL assert the scope query parameter contains exactly `Mail.ReadWrite MailboxSettings.Read offline_access User.Read` (URL-encoded), no more.

### SR-07: Outbound TLS only and egress allowlist

**Threat.** Sift makes outbound HTTPS calls to Microsoft Graph and to four LLM providers. A misconfiguration, a compromised dependency, or a typo'd endpoint URL could redirect traffic to an attacker-controlled host that captures tokens or message metadata. Sift also runs in environments where the user has a corporate proxy or a host firewall and needs to know exactly which destinations to allowlist. Sift exposes no inbound listener in v1, but a future unintended service (a debug HTTP endpoint added during development) could open the host to the LAN if the constraint is implicit instead of explicit.

**Control.** All outbound HTTPS clients (the Graph client and the four vendored router adapters) SHALL be constructed with TLS 1.2 minimum, certificate validation enabled, and no opt-out switch in v1 configuration. The expected egress destinations SHALL be enumerated in `docs/operations.md` as a copy-paste allowlist for corporate proxies and host firewalls: `graph.microsoft.com:443`, `login.microsoftonline.com:443` (OAuth), `api.githubcopilot.com:443` and `api.individual.githubcopilot.com:443` (Copilot), `api.anthropic.com:443` (Claude), `*.openai.azure.com:443` (Azure OpenAI, plus the user's specific deployment host), and `*.cognitiveservices.azure.com:443` plus `*.services.ai.azure.com:443` (Foundry). Sift SHALL NOT bind any inbound socket in v1; a CI policy check SHALL grep the source tree for `socket.bind`, `uvicorn.run`, `httpx.AsyncClient(transport=...)`, `asyncio.start_server`, and any FastAPI / Starlette imports outside `vendor/`, and SHALL fail on positive matches.

**Verification.** A unit test SHALL construct each adapter and the Graph client and SHALL assert the underlying `httpx.AsyncClient` has `verify=True` and `http2=True` (or default verify for stdlib transports) and that a test against a self-signed cert raises `ssl.SSLCertVerificationError`. The CI policy check SHALL run on every PR and SHALL be visible in the PR status as `egress-policy`. The documented allowlist in `docs/operations.md` SHALL match the destinations referenced by the source code, enforced by a test that imports each adapter, reads its base URL, and asserts the host substring appears in the documentation file.

### SR-08: SQLite file permissions

**Threat.** The SQLite state store contains the run history, the undo journal, the backfill checkpoints, the sender cache, and the usage log. The undo journal is particularly sensitive because it links sender addresses to mailbox folder structure and to per-message classification decisions. A world-readable database lets a co-tenant on the host extract the user's mailbox topology and a partial mailing-list inventory without ever touching Microsoft Graph.

**Control.** The state store file (default `~/.local/share/sift/state.db`) SHALL be created with file mode 0600 and parent directory mode 0700. Both modes SHALL be verified on every connection open via `sift.security.permissions.assert_locked_down`, which SHALL refuse to open the file at a wider mode and SHALL exit with code 3 and a remediation message. SQLite write-ahead log files (`state.db-wal`, `state.db-shm`) inherit the database file's mode by default; the helper SHALL also verify those files when present. On Windows, where POSIX mode is not enforced by the file system, the helper SHALL apply ACL checks (no entries other than the current user) via `pywin32` and SHALL refuse to open if any other principal has read or write access.

**Verification.** A unit test SHALL create a database file at 0644 and SHALL assert the helper raises and the process exits with code 3. A second test SHALL create the file fresh, SHALL assert mode 0600 immediately after creation, SHALL touch the WAL companion at 0644 between connections, and SHALL assert the next open raises. The Windows test variant SHALL grant `Everyone:R` to the file via `icacls` and SHALL assert refusal with the equivalent error.

### SR-09: Configuration validation at boundary

**Threat.** Configuration is the largest user-controlled input surface in Sift: YAML files, environment variables, and CLI flags. Without strict validation at the boundary, a typo in a regex rule can be deferred to runtime and surface only when a specific message arrives; a numeric out-of-range value can produce unsafe behaviour (concurrency 1000 against Graph) before any code notices; an unknown YAML key can hide a subtle override (a deprecated key spelled the new way that nobody implemented). Validation done lazily inside business logic is also harder to audit than validation done by a single Pydantic pass at startup.

**Control.** All configuration sources SHALL be loaded into a single Pydantic v2 `Settings` model at startup. Every field SHALL declare its type, default, and validation constraints (`Field(ge=, le=, min_length=, pattern=)` as appropriate). Unknown YAML keys SHALL produce a WARNING with the offending key path and SHALL NOT halt validation. Invalid types, out-of-range values, and structural errors SHALL produce a hard validation error reported as a structured Pydantic error list, exit code 1, and SHALL halt the process before any Graph or LLM call is made. Regex patterns supplied by the user (rule layer, reference-keeper extensions) SHALL be compiled at validation time, not at runtime; a `re.error` at compile SHALL surface as a Pydantic validation error pointing at the YAML location of the bad pattern. The `ConfigValidationError` exception SHALL include the YAML file path, the line number where Pydantic locates the failure when possible, and the offending input value (after redaction per SR-04).

**Verification.** A unit test SHALL feed a YAML with `graph: {concurrency: 200}` and SHALL assert exit code 1 with a Pydantic error referencing `graph.concurrency`. A second test SHALL feed a regex `[unclosed` and SHALL assert the validator surfaces the `re.error` at config load time, not at first message. A third test SHALL feed an unknown key `graph.fancyflag: true` and SHALL assert a WARNING is emitted and validation succeeds. The full property-based test (`hypothesis`) SHALL generate random YAML documents and SHALL assert the loader either accepts and produces a valid `Settings` or rejects with a structured error and exit 1; the loader SHALL never raise an unexpected exception class.

### SR-10: No hard-delete in v1

**Threat.** Hard-delete is irreversible. An LLM hallucination, a regex misfire, or an undo bug that crosses with a hard-delete code path can destroy mail the user wanted to keep, with no recovery short of restoring from backup (and Microsoft 365 personal plans do not give the user the recovery tooling that enterprise plans do). Sift's value proposition depends on user trust; any v1 with a hard-delete path is a v1 that cannot be recommended to anyone with a backlog they cannot easily replace.

**Control.** Excluding the explicit `sift purge --quarantine` command (FR-36, interactive confirmation, no `--yes` flag in v1), no Sift code path SHALL issue a Graph DELETE on a message. All "delete" routing decisions from the rule layer, the reference-keeper, the sender cache, and the LLM SHALL move the message to the configured `Triage/To-Be-Deleted` folder, and from there the user (not Sift) is responsible for confirming purge after the configurable retention period. The Graph client SHALL expose only `move_messages` and `mark_read` methods publicly; a private `_delete_message` SHALL be used only inside the purge command and SHALL be gated by a runtime check that asserts the call site is the purge command. A CI policy check (`ripgrep`) SHALL fail the build if any source file outside `sift/cli/purge.py` references the Graph DELETE verb or the `_delete_message` method.

**Verification.** A unit test SHALL invoke every classifier path with a payload that targets `To-Be-Deleted` and SHALL assert the Graph mock saw zero DELETE calls and exactly one MOVE call to the quarantine folder ID. A CI policy check SHALL grep `DELETE /me/messages` and `_delete_message` outside the purge module and SHALL fail on any match. The purge command's TTY-required behaviour (FR-36) SHALL be tested by piping `/dev/null` to stdin and asserting exit code 5 with the no-TTY diagnostic; the success path SHALL be tested by feeding the typed confirmation phrase. This satisfies success criterion 6 in chunk 1 Section 1.

### SR-11: Vendored router pinning

**Threat.** The vendored `vendor/llm_router/` snapshot is a copy of code from Project Solace at a specific commit. Without an explicit pin, the snapshot can drift silently: a developer updates the vendored code by hand to fix a bug, the change ships, and a downstream Sift deployment then exhibits behaviour that does not match any Solace commit anyone can name. Drift also confuses the friction-log feedback loop that motivates vendoring (the data is only useful if "what we ran" is exactly identifiable).

**Control.** `VENDORED.md` SHALL record (a) the upstream repo URL, (b) the upstream commit SHA, (c) the date of the snapshot, (d) the SHA-256 hash of a deterministic concatenation of every file under `vendor/llm_router/`, and (e) a list of any modifications applied (which SHALL ideally remain empty; see chunk 1 vendoring contract). A CI check `vendor-pin` SHALL recompute the SHA-256 of `vendor/llm_router/` on every PR and SHALL fail if the value disagrees with `VENDORED.md`, unless the same PR updates `VENDORED.md` to match. The vendored adapter test suite (Solace's own tests, copied alongside the snapshot) SHALL run as part of Sift's CI.

**Verification.** A CI test SHALL execute the SHA-256 computation and SHALL assert byte-for-byte equality with `VENDORED.md`. A separate CI test SHALL clone the upstream repo at the recorded SHA, recompute the same hash over the matching subtree, and SHALL assert the local snapshot matches the upstream tree at the pinned SHA (this catches accidental in-place edits). The vendored adapter test suite SHALL pass as part of every Sift CI run. A nightly job MAY check the upstream repo for new SHAs to surface vendoring debt; this SHALL inform but SHALL NOT auto-update.

### SR-12: Dependency provenance

**Threat.** A malicious package version, a typo-squat in `pyproject.toml`, or a high-severity CVE in a transitive dependency can give an attacker code execution inside Sift with full mailbox scope. Lock-file drift (different developers resolving different transitive trees) makes "works on my machine" the only review tool available and turns dependency updates into a guessing game.

**Control.** All Python dependencies SHALL be declared in `pyproject.toml` with version pins or compatible-release specifiers and SHALL be resolved by `uv` into a `uv.lock` checked into the repository. CI SHALL run `uv lock --check` on every PR and SHALL fail if the lock file is out of date relative to `pyproject.toml`. CI SHALL run `pip-audit --strict --requirement <(uv export --no-hashes)` on every PR and SHALL fail on any high or critical advisory in any direct or transitive dependency; medium and low advisories SHALL produce an annotated comment but SHALL NOT block merge. Dependency updates SHALL be applied as discrete PRs (one logical group per PR) so review surface is small. The vendored router (`vendor/llm_router/`) SHALL inherit Sift's dependency policy: all packages it imports SHALL appear in Sift's `pyproject.toml` and SHALL be lock-resolved.

**Verification.** A CI check `lock-fresh` SHALL run `uv lock --check` and SHALL surface its exit code in the PR status. A CI check `audit` SHALL run `pip-audit` and SHALL surface its findings. A separate test SHALL parse the vendored router's source tree to extract its `import` statements and SHALL assert each top-level package appears in `uv.lock`; this catches a vendored router that depends on something Sift does not declare. A periodic (weekly) job SHALL re-run `pip-audit` against `main` and SHALL open an issue when new advisories appear in the resolved set.

### SR-13: Concurrency, throughput, and quota guardrails

**Threat.** Without ceilings, a runaway Sift run can self-DoS by exhausting Microsoft Graph throttling, depleting an LLM provider quota mid-day, or burning a Copilot daily allowance in the first ten minutes. A bug in pagination or in the backfill resume logic can also turn one missing checkpoint into a re-classification of the entire inbox at full provider cost. Operator surprise is itself a security event: a user who blows past their monthly Anthropic spend by 10x because Sift had no per-run cap will rationally never trust the tool again.

**Control.** Sift SHALL enforce three independent ceilings, each settable in configuration with safe defaults, and SHALL refuse to start a run that exceeds any of them: (a) Graph concurrency at most 16 in-flight requests (`graph.concurrency`, default 4, hard upper bound 16, FR-05); (b) per-run message ceiling via `--max-messages` on `triage` and `backfill` (default unset; backfill SHALL also honour `backfill.max_messages` from configuration, FR-30); (c) per-provider per-run token ceilings (`llm.max_prompt_tokens_per_run`, `llm.max_completion_tokens_per_run`, defaults sized to a 1,000-message run at typical token counts; when reached, Sift SHALL stop the run cleanly with a final checkpoint and exit code 6). The vendored cost tracker's `cost_usd` aggregate SHALL also be checked against a soft ceiling (`llm.max_cost_usd_per_run`, default unset); when set and reached, the same clean-stop path SHALL apply. Documentation in `docs/operations.md` SHALL show concrete examples for each ceiling.

**Verification.** A unit test SHALL set `graph.concurrency=20` and SHALL assert config validation fails with exit code 1. A second test SHALL run a fixture-backed `triage --max-messages=5` against an inbox of 100 messages and SHALL assert exactly 5 messages were processed and a final checkpoint was written. A third test SHALL configure `llm.max_completion_tokens_per_run=100` and a fixture LLM that returns 30-token responses and SHALL assert the run stops cleanly after 3 calls with exit code 6 and a stdout summary naming the breached ceiling.

### SR-14: Per-provider credential isolation

**Threat.** When Sift offers four providers but the user has configured only one, two failure modes are unsafe: (a) Sift attempts to call an unconfigured provider, fails with a network or auth error, and writes a confusing log line that leaks details about which provider was attempted; (b) a misconfigured fallback path silently routes mailbox metadata through a provider the user never intended to send data to. Credential bleed between providers (one provider's adapter accidentally reading another provider's environment variable) is a related risk that emerges from convenience helpers.

**Control.** Each adapter SHALL implement `is_configured() -> bool` (per Solace router contract; see chunk 2 component spec for the adapter interface). The implementation SHALL inspect only that adapter's own credentials (its own environment variables, its own keychain entries, its own optional cache file) and SHALL return `False` without raising when its credentials are absent or malformed. The dispatch layer SHALL exclude `is_configured() == False` adapters from the routing pool at startup and SHALL log a single INFO line per excluded provider naming the missing credential (by environment variable name, never the value). When the only configured provider in the pool is the active provider per `llm.provider`, that is the routing pool of one and Sift proceeds. When the active provider is `is_configured() == False`, Sift SHALL exit at startup with code 3 and a remediation message naming the missing credential. There SHALL be no silent fallback to a different provider in v1 (per chunk 1 NFR-08 and FR-23). Each adapter's credential reads SHALL be confined to a per-adapter helper (no shared "get any provider key" function).

**Verification.** A unit test for each of the four adapters SHALL clear all environment variables and SHALL assert `is_configured()` returns `False` without raising. A second test SHALL set only the Claude variables and SHALL assert dispatch's startup pool contains only the Claude adapter, with INFO log lines naming the absent variables for the other three. A third test SHALL set `llm.provider=anthropic` with no Claude credentials and SHALL assert exit code 3 with a remediation message naming `ANTHROPIC_API_KEY`. A static analysis test SHALL parse each adapter file's AST, SHALL collect every `os.environ`, `os.getenv`, and `keyring.get_password` call, and SHALL assert each adapter only references its own credential names (the GitHub Copilot adapter SHALL NOT reference `ANTHROPIC_API_KEY`, etc.).

Where an error condition supports a security control from Section 7, the SR identifier is named in the Recovery column.
